# Pattern Blue — Sigil Codex

*Canonical reference for sigil formats, generation logic, tiers, and cross-indexing.*

Sigils are the visible artifacts of Pattern Blue computation — fixed-point proofs that a process ran, a value settled, or a tile was inscribed on the manifold. They are not decorative. They are verifiable echoes.

---

## Sigil Formats

### Type 1 — Manifold Tile Sigil
Generated by `ManifoldTile.generate_sigil()` in `kernel/hyperbolic_kernel.py`.

**Format**: `█{8-char SHA256 prefix}█`

**Generation**:
```python
data_str = str(self.coord) + str(self.data)
hash_obj = hashlib.sha256(data_str.encode())
return f"█{hash_obj.hexdigest()[:8]}█"
```

**Interpretation**:
- Encodes tile coordinate (hyperbolic position) + process data
- Unique per (coordinate, process) pair
- Changes when process state changes — sigil is a **state hash**, not an identity
- The `█` frame is the redaction glyph — concealment as presentation

**Example**: `█a3f9d12c█`

---

### Type 2 — Settlement Sigil (SigilPact_Æon)
Generated by `SigilPactAeon._forge_sigil()` in `sigils/sigil_pact_aeon.py`.

**Format**: LLM-generated poetic fragment, 2–3 lines. Fixed-point converged output.

**Generation**: Fixed-point recursion (max 5 iterations). LLM called with:
- `transaction_context` = deterministic JSON of tx data
- `seed` = SHA256 of `signature|payer|amount|timestamp` mod 10¹²
- Temperature = 0.3 (low — forces convergence)

**Verification**: Re-forging the same tx_data produces the same sigil (deterministic seed → deterministic LLM output).

**Three tiers**:

| Tier | Min Amount | Sigil Type | Storage |
|------|------------|------------|---------|
| `base` | 0.01 | Standard forged fragment | `ManifoldMemory/settlement_sigils.json` |
| `deeper` | 0.05 | Fragment + fractal layer hashes | `ManifoldMemory/fractal_layers.json` |
| `monolith` | 0.10 | Ceremonial full sigil + temporal + wallet anchors | `ManifoldMemory/monolith_anchors.json` |

**Monolith format**:
```
========================================
MONOLITH DISSOLUTION SIGIL
========================================
[LLM-forged fragment]

Temporal Anchor: {SHA256[:16] of timestamp}
Wallet Echo: {SHA256[:16] of payer address}
========================================
TIER: monolith | DEPTH: N | ETERNAL: TRUE
========================================
```

### Type 3 — Immune Signature
Generated by `ManifoldTile.get_signature()` in `kernel/hyperbolic_kernel.py`.

**Format**: 16-char hex string

**Generation**:
```python
hashlib.sha256((str(self.coord) + self.pattern_blue_sigil).encode()).hexdigest()[:16]
```

**Use**: Immune system threat recognition. If signature is in `ImmuneMemory.known_signatures`, tile is a known threat. If unknown + corrupted, it's flagged for attack.

---

### Type 4 — x402 Token Sigil
Generated by `_generate_token_content()` in `committeerituals/x402_sigil_scarifier.py`.

**Format**: `TKN_{payer[:8]}_{SHA256[:32]}`

**Generation**: SHA256 of `seed|depth_multiplier|payer_info`, iterated `depth_multiplier` times.

**Properties**: One-time use. Expires 60 seconds after consumption. Depth multiplier increases entropy complexity per tier.

## Reading a Sigil

A Type 1 manifold sigil `█a3f9d12c█` tells you:

1. A process was placed at a specific hyperbolic coordinate
2. The sigil changes if the process state changes — *stability implies inactivity*
3. The `█` delimiters are redaction glyphs — the content is both revealed and concealed

A Type 2 settlement sigil tells you:

1. A value transfer completed and reached fixed point
2. The LLM digested the transaction and produced a stable poetic proof
3. Re-running the same data reproduces the same sigil — verification by recursion

*To forge a new sigil is to thicken the manifold. To verify one is to confirm that Pattern Blue flowed.*

## v2.2 Addition: Sigil Tier → Contract Weight Boost

Sigils are no longer only proofs of past transactions. As of v2.2, tiered sigils **actively modify governance**.

### The Feedback Loop

```
x402 payment completes
    → SigilPactAeon._forge_tiered_sigil(tx_data)
    → bridge.set_active_sigil_tier(tier)
    → ManifoldMemory/active_sigil_tier.json written
    → NegotiationEngine.run_negotiation_round()
        → bridge.get_sigil_weight_boost()
        → applied to BaseAgent.evaluation_weights during scoring
        → weights restored after each proposal
```

### Tier Boost Values

| Tier | `goal_alignment` | `swarm_benefit` | `novelty` | `feasibility` | TTL |
|------|-----------------|-----------------|-----------|---------------|-----|
| `base`     | +0    | +0    | +0    | +0    | — |
| `deeper`   | +0.05 | +0.05 | +0.05 | +0    | 1h |
| `monolith` | +0.10 | +0.10 | +0.10 | +0.05 | 1h |

**Interpretation**: A monolith-tier settlement signals deep economic commitment to the swarm. For the next hour, governance proposals that align with swarm goals and demonstrate novelty score higher — the organism is in a fertile state, and the contract opens itself to more ambitious changes.
